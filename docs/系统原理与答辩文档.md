# Counter-Strike 数据库浏览器 — 系统原理与答辩文档

> 面向小组成员与课程答辩。涵盖：技术栈、整体架构、数据库设计、核心原理逐项详解、
> 文件职责清单、扩展方式，以及**模拟老师提问的完整问答**。

---

## 0. 一句话概述

这是一个 **Java Swing 桌面程序 + Oracle 数据库** 的电竞数据管理系统。它能浏览、搜索、
**增/删/改** 11 张表的数据，并提供一组**预置的复杂分析查询**（多表连接、聚合、子查询）。
程序采用**分层架构**，并用**元数据驱动**的方式实现通用 CRUD——也就是说，11 张表共用同一套
增删改查代码，而不是给每张表各写一遍。

## 1. 技术栈

| 层 | 技术 | 说明 |
|---|---|---|
| 界面 | Java **Swing** + **FlatLaf 3.4.1** | FlatLaf 提供现代深色主题（CS2 风格） |
| 业务/数据访问 | 原生 **JDBC** | 通过 `ojdbc17` 驱动连接 Oracle |
| 数据库 | **Oracle Database Free 23**（Docker 容器 `gvenzl/oracle-free`） | 服务名 `FREEPDB1` |
| 构建 | **Maven** + **maven-shade-plugin** | 打成可直接运行的 fat jar（约 8.2 MB，含所有依赖） |
| 运行环境 | 编译目标 Java 17，运行在 JDK 21 | |

**为什么这样选型？**
- 用原生 JDBC 而非 ORM（如 Hibernate），是为了在课程中**直接看到并掌握 SQL**——这正是数据库课的核心。
- 用 Docker 跑 Oracle，避免本机装大型数据库的复杂安装，团队环境一致、可复现。
- fat jar 让程序"一个文件即可运行"，方便演示。

## 2. 整体架构（分层）

程序严格分四层，**上层只依赖下层、互不越级**，这叫"关注点分离"（Separation of Concerns）：

```
┌─────────────────────────────────────────────────────────┐
│  表示层 (UI)        Swing 界面                            │
│  MainFrame · CrudPanel · AnalyticsPanel ·                │
│  ReadOnlyTableModel · Styles                             │
└───────────────┬─────────────────────────────────────────┘
                │ 调用方法 / 传 DTO（TableData 等）
┌───────────────▼─────────────────────────────────────────┐
│  数据访问层 (Repository)  所有 SQL 都在这里               │
│  CounterStrikeRepository                                 │
│  + 元数据：ColumnSpec / TableSpec / Schema               │
│  + 查询目录：AnalyticsQuery / AnalyticsCatalog           │
└───────────────┬─────────────────────────────────────────┘
                │ 借用 Connection
┌───────────────▼─────────────────────────────────────────┐
│  连接层 (DB)        Database（连接工厂）                  │
│  DatabaseConfig（读 app.properties）                     │
└───────────────┬─────────────────────────────────────────┘
                │ JDBC
┌───────────────▼─────────────────────────────────────────┐
│  Oracle Database (FREEPDB1)  11 张表                     │
└─────────────────────────────────────────────────────────┘
```

**启动数据流**（`CounterStrikeApp.main`）：
1. `FlatDarkLaf.setup()` 安装深色主题。
2. `DatabaseConfig.load()` 读取 `app.properties`（URL / 用户 / 密码）。
3. `new Database(config)` 建连接工厂，`testConnection()` 先验证能连上。
4. `new CounterStrikeRepository(database)` 建数据访问层。
5. `new MainFrame(repository, url)` 建主窗口并显示。
6. 任一步出错 → 弹出错误对话框（不会白屏崩溃）。

**分层的好处（答辩常考）**：界面改动不影响 SQL；SQL 改动不影响界面；
数据库换成 MySQL 也只需改连接层；每层都可单独理解和测试。

## 3. 数据库设计

### 3.1 11 张表与关系

| 表 | 主键 | 说明 | 主要外键 |
|---|---|---|---|
| **People** | PersonID | 所有人（选手、教练）的公共信息 | — |
| **Player** | PersonID | 选手专有属性（Rating/ADR/DPR） | PersonID → People |
| **Coach** | PersonID | 教练专有属性（WinRate） | PersonID → People |
| **Team** | TeamName | 战队 | — |
| **PlayerContract** | PlayerContractID | 选手合同 | PersonID→Player, TeamName→Team |
| **CoachContract** | CoachContractID | 教练合同 | PersonID→Coach, TeamName→Team |
| **Tournament** | TournamentID | 赛事 | — |
| **MatchTable** | MatchID | 比赛（BO 级别） | TournamentID, TeamA, TeamB, WinnerTeam |
| **MatchRecord** | MatchRecordID | 单图记录 | MatchID, TeamA, TeamB |
| **PlayerMatchStats** | (MatchRecordID, PlayerID) | 单图选手数据 | 三个外键，**复合主键** |
| **TournamentParticipation** | (TournamentID, TeamName) | 参赛与名次 | 两个外键，**复合主键** |

### 3.2 关键设计点

- **ISA / 泛化-特化（继承）**：`People` 是父类，`Player` 和 `Coach` 通过**共享主键**
  （PersonID 同时是主键又是指向 People 的外键）实现"一个人可以是选手或教练"。
  这是 ER 模型里 ISA 层次的经典关系型实现方式。
- **复合主键**：`PlayerMatchStats` 用 (MatchRecordID, PlayerID) 作主键——表示"某图某选手"
  这条记录唯一；`TournamentParticipation` 用 (TournamentID, TeamName)。
- **外键完整性**：比赛的 TeamA/TeamB/WinnerTeam 都必须是已存在的队伍，由外键保证。
- **删除策略**：外键**没有**用 `ON DELETE CASCADE`，而是默认的 **RESTRICT**（有子记录就不许删）。
  我们在程序里把这个错误（ORA-02292）翻译成友好提示，让用户先删子记录——更安全，避免误删一整串数据。

### 3.3 范式

表设计基本满足 **第三范式（3NF）**：
- 每个属性都依赖于主键（无部分依赖、无传递依赖）。
- 选手/教练的专有属性被拆到独立表，避免 People 表出现大量 NULL。
- 合同、参赛、单图数据都独立成表，消除了重复存储。

## 4. 核心原理逐项详解

### 4.1 配置加载（`DatabaseConfig`）
- 用 `record` 存 url/user/password，**不可变**。
- 按优先级查找配置：`-Dcs2.config` 系统属性 → `CS2_DB_CONFIG` 环境变量 → 项目根目录
  `app.properties` → classpath。找不到必填项就抛清晰异常。
- 好处：**密码等敏感信息不写死在代码里**，团队各自配置，也不进 Git。

### 4.2 连接管理（`Database` + try-with-resources）
```java
try (Connection connection = database.getConnection();
     PreparedStatement statement = connection.prepareStatement(sql);
     ResultSet resultSet = statement.executeQuery()) {
    ...
} // 离开 try 块自动 close()，即使抛异常也会关
```
- 每次操作**新建连接、用完即关**，靠 try-with-resources 保证不泄漏。
- （可改进点：生产环境应加**连接池**，见 §8。）

### 4.3 防 SQL 注入：一律用 `PreparedStatement` + `?` 占位符
**绝不**用字符串拼接把用户输入塞进 SQL。所有用户值都通过 `setString/setLong/...` 绑定：
```java
String sql = "INSERT INTO Team (TeamName, Region) VALUES (?, ?)";
statement.setString(1, teamName);   // 即使输入 ' OR 1=1 -- 也只是普通字符串
statement.setString(2, region);
```
- 数据库把 `?` 当**数据**而非**代码**，从根本上杜绝 SQL 注入。
- 附带好处：相同 SQL 可被数据库**预编译并复用执行计划**，性能更好。
- 注意：**表名/列名不能用占位符**，所以这些只来自我们自己定义的 `Schema` 元数据（白名单），
  绝不来自用户输入——同样安全。

### 4.4 元数据驱动的通用 CRUD（**本项目最核心的设计**）

不给 11 张表各写一套增删改查，而是**用对象描述表结构**，再写一套通用代码读这些描述。

**三个元数据类：**
- `ColumnSpec`：描述一列 — 列名、显示名、类型(TEXT/INTEGER/DECIMAL/DATE)、是否主键、是否必填、是否外键。
- `TableSpec`：描述一张表 — 表名、列清单、排序方式；并能算出主键列、可编辑列等。
- `Schema`：把全部 11 张表的 `TableSpec` 集中定义在一处。

**通用 SQL 是"拼"出来的**（用元数据动态生成，但值仍走占位符）：
```java
// insertRow：根据列清单生成 INSERT ... VALUES (?,?,...)
String sql = "INSERT INTO " + spec.tableName() + " (" + names + ") VALUES (" + marks + ")";
// 再按每列类型把表单文本转成正确的 JDBC 类型后绑定
bind(statement, index, column, rawValues.get(column.name()));
```
- `browse()`：`SELECT 所有列 FROM 表 [WHERE 每列 LIKE ?] ORDER BY ...`，实现全列模糊搜索。
- `insertRow / updateRow / deleteRow`：分别拼 INSERT / UPDATE…WHERE 主键 / DELETE…WHERE 主键。
- 复合主键自动支持（主键列可能不止一个，WHERE 用 `AND` 连接）。

**界面也是元数据驱动的**：`CrudPanel` 读 `TableSpec` 自动**生成表单**——
普通列给文本框，外键列给**下拉框**（选项来自被引用表的 `distinctValues`），日期列提示 `YYYY-MM-DD`。

> **价值**：要加一张新表或一个新字段，**只改 `Schema.java` 一处**，界面和增删改查全部自动生效。
> 这就是"数据驱动/配置驱动"思想，大幅减少重复代码和出错点。

### 4.5 事务与 ACID（写操作的安全保证）
所有写操作（增/改/删）都包在显式事务里：
```java
connection.setAutoCommit(false);     // 关闭自动提交
try {
    ... executeUpdate ...
    connection.commit();             // 成功才提交
} catch (SQLException e) {
    connection.rollback();           // 失败回滚，数据恢复原状
    throw 友好异常;
}
```
- 保证操作的**原子性**：要么全成功，要么全不改，不会留下半成品数据。
- 这是数据库 **ACID** 中 A(原子性) 和 C(一致性) 在应用层的体现。

### 4.6 外键约束 + 友好错误翻译
当违反约束时 Oracle 抛带错误码的异常，我们翻译成人能看懂的提示：

| Oracle 错误码 | 含义 | 我们的提示 |
|---|---|---|
| ORA-00001 | 主键/唯一键重复 | 记录已存在 |
| ORA-02291 | 外键父记录不存在 | 请先创建父记录 |
| ORA-02292 | 有子记录引用，不能删 | 请先删除依赖记录 |
| ORA-01400 | 必填字段为空 | 必填项未填 |
| ORA-01438 / 12899 | 值超出列长度 | 值过长 |

### 4.7 多线程：SwingWorker + EDT（界面不卡死）
Swing 有"**事件分发线程（EDT）**"规则：所有界面更新必须在 EDT 上，但**耗时操作（查数据库）绝不能在 EDT 上**，否则界面冻结。
```java
new SwingWorker<TableData, Void>() {
    protected TableData doInBackground() { return repository.browse(...); } // 后台线程查库
    protected void done() { 更新表格; }                                    // 回到 EDT 更新界面
}.execute();
```
- `doInBackground()` 在后台线程执行查询，`done()` 自动回到 EDT 刷新界面 → **查询时界面依然流畅**。

### 4.8 高级查询（`AnalyticsCatalog` 的 8 个预置报表）
展示了数据库课要求的各种 SQL 能力：

| 报表 | 用到的 SQL 技术 |
|---|---|
| 选手赛事表现排行 | 多表 JOIN + `GROUP BY` + 聚合(`AVG/SUM/COUNT`) + `HAVING` |
| 战队胜负战绩 | `LEFT JOIN` + `CASE WHEN` 条件计数 + `NULLIF` 防除零 |
| 战队当前阵容 | JOIN + 日期条件（活跃合同：EndDate 为空或在未来） |
| 超均值评分选手 | **子查询**（`WHERE Rating > (SELECT AVG(Rating) ...)`） |
| 赛事参赛名次 | 多表 JOIN + 排序 |
| 地图统计 | `GROUP BY` + `COUNT` + `MAX` |
| 教练胜率排行 | JOIN + 排序 + `NULLS LAST` |
| 战队对战记录 | JOIN + 带参数的 `LIKE` 过滤（用户输入队名） |

- 参数化查询通过 `?` 绑定（同样防注入）；结果列名用 `ResultSetMetaData` 动态读取，所以结果表格能自适应任意查询。

## 6. 如何扩展（给小组成员）

**加一个字段**：在 `Schema.java` 对应表的列清单里加一行 `col("新列名","显示名",类型)`，重新编译即可——浏览、表单、增删改全部自动支持。

**加一张表**：在 `Schema.java` 新建一个 `TableSpec` 并加进 `Schema.ALL`，管理页下拉框里就会出现它。

**加一个分析报表**：在 `AnalyticsCatalog.QUERIES` 里加一个 `AnalyticsQuery`（写好 SQL 即可），高级查询页自动出现。

## 8. 已知可改进点（主动说明，体现工程素养）

1. **连接池**：现在每次操作新建连接。生产应引入 HikariCP 等连接池复用连接，降低开销。
2. **分页**：`browse` 一次取全表，数据量极大时应做分页（Oracle `OFFSET ... FETCH`）。
3. **乐观锁**：多人同时改同一行时没有版本控制，后写覆盖先写。可加版本列检测冲突。
4. **审计日志**：未记录"谁在何时改了什么"。
5. **外键下拉显示**：外键下拉目前显示 ID（如 PersonID），可优化为"ID - 姓名"更友好。

# 附：模拟老师答辩问答

### A. 架构类

**Q：为什么要分这么多层？直接在按钮事件里写 SQL 不行吗？**
A：能跑但难维护。分层后界面、SQL、连接各管各的：改界面不碰 SQL，换数据库只改连接层，
每层可单独测试。这是"关注点分离"，也降低了耦合。

**Q：Repository 层的作用？**
A：它是**唯一**写 SQL 的地方，把数据库操作封装成方法（如 `browse`、`insertRow`）。
界面只调方法、拿结果，完全不知道底层是 Oracle 还是别的库。

**Q：DTO（TableData）是干什么的？**
A：在层之间传数据的简单容器。Repository 把查询结果打包成 `TableData`（列名+行）交给界面，
界面不需要直接接触 `ResultSet`，解耦更彻底。

### B. 数据库理论类

**Q：你的表满足第几范式？**
A：第三范式（3NF）。每个非主属性都直接依赖主键，没有部分依赖和传递依赖。
比如把选手专有属性放进 Player 表、教练的放 Coach 表，避免 People 表大量 NULL 和冗余。

**Q：People、Player、Coach 是什么关系？**
A：ISA（泛化/特化，即继承）。People 存公共信息，Player/Coach 存专有信息，
它们**共享主键** PersonID——子表的主键同时是指向 People 的外键。
这样"一个人是选手还是教练"由是否在对应子表有记录决定。

**Q：复合主键在哪里用到？为什么？**
A：`PlayerMatchStats` 用 (MatchRecordID, PlayerID)、`TournamentParticipation` 用
(TournamentID, TeamName)。因为单独一列不能唯一确定一行——"某图的某个选手"才唯一，
"某赛事的某支队伍"才唯一。

**Q：什么是 ACID？你的程序怎么体现？**
A：原子性、一致性、隔离性、持久性。我的写操作用 `setAutoCommit(false)` + `commit/rollback`：
要么全部成功提交，要么出错全部回滚——保证了原子性和一致性。隔离性和持久性由 Oracle 本身保证。

**Q：为什么外键不用 ON DELETE CASCADE？**
A：级联删除会"删一行带走一整串"，演示或误操作时很危险。我选择默认的 RESTRICT——
有子记录就不让删，并在程序里把这个错误翻译成"请先删依赖记录"，更安全可控。

### C. SQL 类

**Q：INNER JOIN 和 LEFT JOIN 区别？你哪里用了 LEFT JOIN？**
A：INNER JOIN 只返回两表都匹配的行；LEFT JOIN 保留左表全部行，右表无匹配则填 NULL。
"战队胜负战绩"用 LEFT JOIN，是为了**让没有任何比赛的队伍也出现在结果里**（场次显示 0）。

**Q：GROUP BY 和 HAVING 的区别？**
A：GROUP BY 按列分组做聚合；WHERE 在分组前过滤行，HAVING 在分组后过滤组。
比如"选手表现排行"里 `HAVING COUNT(*)>=1` 是对聚合结果过滤。

**Q：子查询用在哪里？**
A："超均值评分选手"：`WHERE Rating > (SELECT AVG(Rating) FROM Player)`。
括号里的子查询先算出全体平均分，外层再筛出高于它的选手。

**Q：NVL / NULLIF / NULLS LAST 是什么？**
A：都是处理 NULL。`NVL(a,b)` 是 a 为空就取 b；`NULLIF(a,b)` 是 a=b 就返回 NULL
（我用它防止除零：`COUNT/NULLIF(总数,0)`）；`ORDER BY ... NULLS LAST` 让空值排在最后。

**Q：怎么算胜率的？**
A：用 `CASE WHEN m.WinnerTeam=t.TeamName THEN 1 ELSE 0 END` 做条件计数得到胜场，
再除以总场数并乘 100，用 NULLIF 防止 0 场时除零，ROUND 保留一位小数。

### D. 安全类

**Q：怎么防 SQL 注入？**
A：所有用户输入都通过 `PreparedStatement` 的 `?` 占位符绑定，数据库把它当数据不当代码。
表名列名不接受用户输入，只来自我们代码里写死的 `Schema` 元数据（白名单）。

**Q：用户在搜索框输入 `' OR 1=1 --` 会怎样？**
A：什么都不会发生。它会被当成一个普通的搜索字符串去 `LIKE` 匹配，匹配不到就返回空，
不会改变 SQL 结构。

### E. Java / 并发类

**Q：为什么用 SwingWorker，不直接查库？**
A：Swing 界面更新必须在 EDT 线程，但在 EDT 上做耗时查询会冻结界面。
SwingWorker 让 `doInBackground` 在后台线程查库、`done` 回到 EDT 刷新，界面始终流畅。

**Q：try-with-resources 解决什么问题？**
A：自动关闭 Connection/Statement/ResultSet，即使中途抛异常也会关，避免连接泄漏。

### F. 设计取舍类

**Q：元数据驱动的 CRUD 有什么优缺点？**
A：优点是**极少重复代码**、加表加字段只改一处、不易出错。
缺点是通用代码比手写的稍抽象、不好读，且对"特别定制的界面/校验"不如手写灵活。
对本项目（11 张结构相似的表）非常划算。

**Q：如果数据量涨到几百万行，你的程序会怎样？怎么改？**
A：现在 `browse` 一次取全表会变慢、占内存。应做**分页**（Oracle `OFFSET n ROWS FETCH NEXT m ROWS ONLY`），
并给常用搜索列**加索引**，再引入**连接池**复用连接。

**Q：为什么不用 Hibernate/JPA 这类 ORM？**
A：课程目标是掌握 SQL 本身。用原生 JDBC 能直接看到、控制每一条 SQL，
更贴合数据库课的学习目的；ORM 会把 SQL 藏起来。
