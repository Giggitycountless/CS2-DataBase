# Counter-Strike 数据库浏览器 — 系统原理与答辩文档

> 面向小组成员与课程答辩（Milestone 1–3 全覆盖）。涵盖：技术栈、整体架构、数据库设计、
> MongoDB 设计、索引性能分析、核心原理逐项详解，以及**模拟老师提问的完整问答**。

---

## 0. 一句话概述

这是一个 **Java Swing 桌面程序 + Oracle + MongoDB** 的电竞数据管理系统。它能浏览、搜索、
**增/删/改** 11 张 Oracle 表的数据，提供一组**预置复杂分析查询**，并可在运行时切换到
MongoDB 后端进行同等读写操作。程序采用**分层架构**，用**元数据驱动**的方式实现通用 CRUD，
并在 Milestone 3 中完成了 B-Tree 索引与 Hash Cluster 的性能分析。

## 3. Oracle 数据库设计

### 3.1 11 张表与关系

| 表 | 主键 | 说明 | 主要外键 |
|---|---|---|---|
| **People** | PersonID | 所有人（选手、教练）公共信息 | — |
| **Player** | PersonID | 选手专有属性（Rating/ADR/DPR） | PersonID → People |
| **Coach** | PersonID | 教练专有属性（WinRate） | PersonID → People |
| **Team** | TeamName | 战队 | — |
| **PlayerContract** | PlayerContractID | 选手合同 | PersonID→Player, TeamName→Team |
| **CoachContract** | CoachContractID | 教练合同 | PersonID→Coach, TeamName→Team |
| **Tournament** | TournamentID | 赛事 | — |
| **MatchTable** | MatchID | 比赛（BO 级别） | TournamentID, TeamA, TeamB, WinnerTeam |
| **MatchRecord** | MatchRecordID | 单图记录 | MatchID, TeamA, TeamB |
| **PlayerMatchStats** | (MatchRecordID, PlayerID) | 单图选手数据 | 三个外键，**复合主键** |
| **TournamentParticipation** | (TournamentID, TeamName) | 参赛与名次 | 两个外键，**复合主键** |

### 3.2 关键设计点

- **ISA / 继承**：`People` 是父类，`Player` 和 `Coach` 共享主键 PersonID（同时是指向 People 的外键），实现 ER 的泛化-特化层次。
- **复合主键**：`PlayerMatchStats(MatchRecordID, PlayerID)` 和 `TournamentParticipation(TournamentID, TeamName)`。
- **外键完整性**：比赛的 TeamA/TeamB/WinnerTeam 均约束为已存在的队伍。
- **删除策略**：外键用默认 **RESTRICT**（有子记录不许删），程序把 ORA-02292 翻译成友好提示。
- **第三范式（3NF）**：每个非主属性直接依赖主键，无部分依赖和传递依赖。

### 3.3 数据集规模

| 表 | 小数据集 | 大数据集 |
|---|---|---|
| People / Player / Coach / Team | 48 / 40 / 8 / 8 | 同左 |
| Tournament | 1 | 200 |
| MatchTable | 7 | 20,200 |
| MatchRecord | 18 | 20,000 |
| PlayerMatchStats | ~60 | **200,000** |

## 4. MongoDB 设计

### 4.1 数据库与集合

数据库名：`cs_small`，包含 4 个集合：

| 集合 | 文档数 | 说明 |
|---|---|---|
| `teams` | 8 | `{ _id: "Team Spirit", region: "CIS" }` |
| `people` | 10 | 人员基本信息，内嵌 playerStats/coachStats 和 contract |
| `tournaments` | 1 | 基本信息，内嵌 participation 数组 |
| `matches` | 7 | 比赛，内嵌 records 数组，每条 record 内嵌 playerStats 数组 |

### 4.2 文档结构示例（people 集合）

```json
{
  "_id": 11,
  "nickname": "donk",
  "fullName": "Danil Kryshkovets",
  "birthday": ISODate("2007-01-25"),
  "nationality": "Russia",
  "type": "player",
  "playerStats": { "rating": 1.32, "adr": 90.4, "dpr": 0.66 },
  "contract": {
    "teamName": "Team Spirit",
    "startDate": ISODate("2023-07-01"),
    "endDate": null,
    "inGameRole": "Rifler"
  }
}
```

### 4.3 MongoDB vs Oracle 设计对比

| 维度 | Oracle（关系型） | MongoDB（文档型） |
|---|---|---|
| 结构 | 11 张规范化表 | 4 个集合，数据内嵌 |
| 人员+合同 | People + Player/Coach + Contract 四张表 JOIN | people 一个文档，stats 和 contract 内嵌 |
| 比赛+成绩 | MatchTable + MatchRecord + PlayerMatchStats 三张表 | matches 一个文档，records 和 playerStats 内嵌 |
| 读单个球员全部信息 | 需要多表 JOIN | 读一个文档，O(1) |
| 范围查询/聚合 | SQL 强大，任意 GROUP BY | 聚合管道，较复杂 |
| 数据一致性 | 外键约束保证 | 应用层负责 |
| 适合场景 | 结构固定、关系复杂、一致性要求高 | 层次数据、读多写少、schema 灵活 |

**为什么这样嵌套？** 应用最常见的查询是"某个球员的全部信息"和"某场比赛的全部成绩"，
把相关数据嵌进同一个文档，避免了 JOIN，读一次文档即可。

## 5. Milestone 3 — 索引性能分析

### 5.1 两条高负载查询

**Q1 — 点查询**：查询某一球员（PlayerID = 11）在所有比赛中的统计数据。
在 PlayerMatchStats（20 万行）上做等值过滤，跨 4 张表 JOIN。

```sql
SELECT p.Nickname, pms.Kills, pms.Deaths, pms.Assists, pms.Rating, pms.ADR,
       mr.Map, mt.Stage, mt.MatchDate
FROM PlayerMatchStats pms
JOIN People      p  ON p.PersonID       = pms.PlayerID
JOIN MatchRecord mr ON mr.MatchRecordID = pms.MatchRecordID
JOIN MatchTable  mt ON mt.MatchID       = mr.MatchID
WHERE pms.PlayerID = 11
ORDER BY mt.MatchDate DESC;
```

**Q2 — 范围查询**：查询某日期区间内所有比赛图的聚合统计。
在 MatchRecord（2 万行）上做 BETWEEN 范围过滤，JOIN PlayerMatchStats 聚合。

```sql
SELECT mr.RecordDate, mr.Map, mr.TeamA, mr.TeamB, mr.FinalScore,
       COUNT(pms.PlayerID) AS PlayerCount,
       AVG(pms.Rating)     AS AvgRating,
       MAX(pms.Kills)      AS TopKills
FROM MatchRecord mr
JOIN PlayerMatchStats pms ON pms.MatchRecordID = mr.MatchRecordID
WHERE mr.RecordDate BETWEEN DATE '2023-01-01' AND DATE '2024-12-31'
GROUP BY mr.RecordDate, mr.Map, mr.TeamA, mr.TeamB, mr.FinalScore
ORDER BY mr.RecordDate;
```

### 5.2 性能测量结果

**Q1 — 点查询（PlayerID = 11）**

| 方案 | EXPLAIN PLAN 关键步骤 | 执行时间 |
|---|---|---|
| 基准（无额外索引） | TABLE ACCESS FULL on PLAYERMATCHSTATS | **0.34s** |
| B-Tree 索引 (`idx_pms_playerid`) | TABLE ACCESS FULL（优化器拒绝使用索引） | **0.17s** |
| Hash Cluster (`pms_pid_cluster`) | TABLE ACCESS HASH on PLAYERMATCHSTATS_HC | **2.92s** |

**Q2 — 范围查询（RecordDate BETWEEN）**

| 方案 | EXPLAIN PLAN 关键步骤 | 执行时间 |
|---|---|---|
| 基准（无额外索引） | TABLE ACCESS FULL on MATCHRECORD | **0.02s** |
| B-Tree 索引 (`idx_mr_recorddate`) | INDEX RANGE SCAN on IDX_MR_RECORDDATE | **0.02s** |
| Hash Cluster (`mr_date_cluster`) | TABLE ACCESS FULL on MATCHRECORD_HC（cost=555 vs 68） | **0.02s** |

### 5.3 分析与讨论

**Q1 B-Tree 索引未被使用**：Oracle 的代价优化器（CBO）计算出每个 PlayerID 约有 5046 行
（占全表 2.5%），判断多块顺序全扫（cost=342）比 5046 次随机 I/O 更划算，因此拒绝使用索引。
这是正确行为——B-Tree 索引在**高选择率（<1%）时最有效**。

**Q1 Hash Cluster 更慢（2.92s）**：`SIZE=8192` 远小于实际每个 PlayerID 的数据量
（5000行×50字节≈250KB），造成大量溢出链，需要读取许多额外块，反而比全扫慢。
Hash Cluster 在 SIZE 设置合理时能做到 O(1) 点查找，但参数需精确估算。

**Q2 B-Tree 索引成功使用**：EXPLAIN PLAN 确认从 TABLE ACCESS FULL 切换为
INDEX RANGE SCAN（cost 从 68 降到 4），**B-Tree 天然有序，非常适合范围查询**。
由于结果集仅 13 条记录，两种方式的挂钟时间相近，但 cost 下降说明 I/O 减少。

**Q2 Hash Cluster 无法范围查询**：哈希函数无序，无法做 BETWEEN，
被迫全表扫描（cost=555，远高于原来 68），印证了 **Hash Cluster 仅适合等值点查询**。

## 6. 核心原理详解

### 6.1 元数据驱动的通用 CRUD（本项目最核心的设计）

不给 11 张表各写一套增删改查，而是用三个元数据类描述表结构，再写一套通用代码：

- `ColumnSpec`：列名、类型(TEXT/INTEGER/DECIMAL/DATE)、是否PK/FK/必填
- `TableSpec`：表名、列清单、主键列
- `Schema`：11 张表的集中定义

通用 SQL 动态拼接，值仍走 `PreparedStatement` 占位符：
```java
String sql = "INSERT INTO " + spec.tableName() + " (" + names + ") VALUES (" + marks + ")";
```

界面也由元数据驱动：`CrudPanel` 读 `TableSpec` 自动生成表单——外键列变下拉框，日期列提示格式。

**价值**：加一张表只改 `Schema.java` 一处，界面和 CRUD 全部自动生效。

### 6.2 PreparedStatement 防 SQL 注入

所有用户输入通过 `?` 占位符绑定，数据库将其视为数据而非代码。
表名/列名仅来自 `Schema` 元数据（白名单），绝不接受用户输入。

### 6.3 事务与 ACID

```java
connection.setAutoCommit(false);
try {
    ... executeUpdate ...
    connection.commit();
} catch (SQLException e) {
    connection.rollback();
}
```
保证写操作**原子性**：要么全成功，要么全回滚。

### 6.4 外键约束 + 友好错误翻译

| Oracle 错误码 | 含义 | 程序提示 |
|---|---|---|
| ORA-00001 | 主键重复 | 记录已存在 |
| ORA-02291 | 外键父记录不存在 | 请先创建父记录 |
| ORA-02292 | 有子记录引用，不能删 | 请先删除依赖记录 |
| ORA-01400 | 必填字段为空 | 必填项未填 |

### 6.5 SwingWorker 防界面冻结

数据库查询在 `doInBackground()` 后台线程执行，`done()` 回到 EDT 刷新界面，
保证查询期间 UI 保持响应。

### 6.6 8 条高级分析查询

| 报表 | SQL 技术 |
|---|---|
| 选手赛事表现排行 | 多表 JOIN + GROUP BY + AVG/SUM/COUNT + HAVING |
| 战队胜负战绩 | LEFT JOIN + CASE WHEN 条件计数 + NULLIF 防除零 |
| 战队当前阵容 | JOIN + 日期条件（EndDate 为空或在未来） |
| 超均值评分选手 | 子查询（WHERE Rating > (SELECT AVG...)） |
| 赛事参赛名次 | 多表 JOIN + 排序 |
| 地图统计 | GROUP BY + COUNT + MAX |
| 教练胜率排行 | JOIN + ORDER BY NULLS LAST |
| 战队对战记录 | JOIN + 参数化 LIKE |

## 8. 已知可改进点

1. **连接池**：现在每次操作新建连接，生产环境应引入 HikariCP。
2. **分页**：`browse` 一次取全表，大数据集应做 `OFFSET … FETCH`。
3. **Hash Cluster SIZE 精确估算**：SIZE 需要根据实际行数和行大小计算，否则溢出链会更慢。
4. **乐观锁**：多人并发写同一行没有版本控制。
5. **连接池**：MongoDB 连接每次新建，可用 MongoClient 的内置池复用。

# 附：模拟答辩问答

## A. 架构类

**Q：为什么分这么多层？**
A：分层后各层职责清晰：界面只管显示，SQL 只在 Repository，连接只在 DB 层。
改界面不碰 SQL，换数据库只改连接层，这叫"关注点分离"，降低耦合，方便维护和测试。

**Q：AppRepository 接口有什么用？**
A：Oracle 和 MongoDB 两个实现都实现同一接口，UI 层只持有接口引用。
切换后端时只需替换实现类，UI 代码一行不改——这是**面向接口编程**（依赖倒置）。

**Q：DTO 是干什么的？**
A：`TableData`（列名+行）是 Repository 和 UI 之间传递数据的简单容器，
UI 不直接接触 `ResultSet`，解耦更彻底。

## B. 数据库理论类

**Q：你的表满足第几范式？**
A：第三范式（3NF）。每个非主属性直接依赖主键，无部分依赖和传递依赖。
例如把选手专有属性放 Player 表、教练的放 Coach 表，避免 People 表出现大量 NULL 和冗余。

**Q：People、Player、Coach 是什么关系？**
A：ISA（泛化/特化，即继承）。People 存公共信息，Player/Coach 存专有信息，
共享主键 PersonID——子表的主键同时是指向 People 的外键。

**Q：复合主键为什么用？**
A：`PlayerMatchStats(MatchRecordID, PlayerID)`——"某图的某个选手"才唯一；
`TournamentParticipation(TournamentID, TeamName)`——"某赛事的某队伍"才唯一。

**Q：什么是 ACID？你怎么体现？**
A：原子性、一致性、隔离性、持久性。我的写操作用 `setAutoCommit(false)` + `commit/rollback`：
成功提交，出错全部回滚，保证了原子性。隔离和持久性由 Oracle 本身保证。

## C. SQL 类

**Q：INNER JOIN 和 LEFT JOIN 区别？哪里用了 LEFT JOIN？**
A：INNER JOIN 只返回两表都匹配的行；LEFT JOIN 保留左表全部行。
"战队胜负战绩"用 LEFT JOIN，让没有任何比赛的队伍也出现在结果里（场次显示 0）。

**Q：子查询用在哪里？**
A："超均值评分选手"：`WHERE Rating > (SELECT AVG(Rating) FROM Player)`。
括号里先算全体平均分，外层再筛出高于它的选手。

**Q：怎么防 SQL 注入？**
A：所有用户输入通过 PreparedStatement `?` 占位符绑定，数据库将其视为数据不当代码。
表名列名只来自我们自己的 Schema 元数据（白名单），绝不来自用户输入。

## D. Milestone 3 — 索引与 MongoDB 类

**Q：B-Tree 索引为什么对 Q1 没有提升？**
A：Oracle 的代价优化器计算出 PlayerID=11 约返回 5046 行（占全表 2.5%），
顺序全扫比 5046 次随机 I/O 更快，所以拒绝使用索引。
B-Tree 索引在高选择率（通常 <1%）时效果最好。

**Q：Hash Cluster 为什么对 Q1 反而更慢？**
A：创建时 `SIZE=8192` 远小于实际每个 PlayerID 的数据量（约 250KB），
导致大量溢出链，需要读取很多额外块，I/O 反而比全扫多。
Hash Cluster 要发挥效果，SIZE 必须精确估算。

**Q：Hash Cluster 为什么不适合范围查询？**
A：哈希函数把键映射到随机桶，没有顺序。要找 `BETWEEN` 区间，
只能全表扫描所有桶逐一判断，比不上 B-Tree 的有序范围扫描。

**Q：MongoDB 和 Oracle 设计最大的区别是什么？**
A：Oracle 是规范化设计，11 张表，关系通过外键连接，读单个球员需要多表 JOIN。
MongoDB 是文档嵌套设计，把球员信息、统计和合同嵌进同一个文档，
读一次文档就能拿到所有数据，但失去了跨集合的约束保证。

**Q：什么情况下选 MongoDB，什么时候选 Oracle？**
A：MongoDB 适合：读多写少的层次型数据、schema 频繁变化、不需要强一致性约束。
Oracle 适合：结构固定、关系复杂、需要事务和外键约束保证数据一致性。
本项目的域（赛事、球员、合同关系复杂）更适合 Oracle；MongoDB 作为扩展展示了不同建模思路。

**Q：为什么 MongoDB 的集合比 Oracle 的表少那么多？**
A：MongoDB 用文档嵌套把相关数据聚合在一起。Oracle 里需要 People + Player + PlayerContract
三张表才能描述"某个球员及其合同"，MongoDB 用一个 people 文档就搞定了。
这减少了集合数量，但也意味着无法用数据库层面的约束保证嵌套数据的完整性。
